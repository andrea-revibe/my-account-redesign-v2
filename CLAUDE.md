# CLAUDE.md

Internal-demo prototype for the Revibe **My Account → Orders** area. Used to evaluate UX/visual changes before specifying them for production. Not a real product.

## Where to work

App code lives at the repo root.

```sh
npm install
npm run dev      # http://localhost:5173 (Vite)
npm run build
```

The mobile frame is **430px wide**. Screenshots, tests, and visual checks should use that viewport.

## Stack

React 19 + Vite 8 · Tailwind 3 · lucide-react · Inter (Graphik substitute)

## Where things live

For architectural rationale (chrome families, why certain choices), start at `docs/README.md` and follow into the per-feature doc. This is just the map.

- `src/App.jsx` — page composition; owns filter state, `claimFlowOrderId`, the in-session `submittedClaims` map (orderId → claim) that's projected over `ORDERS` so a freshly-submitted claim flips the order to its claim-card variant immediately, and `journeyMode` + `journeyId` state from the `?journey=<id>` URL param. In journey mode, `ORDERS` is replaced by the single replayed order from `useJourney(journeyId)`. Routes orders to one of eight cards (see Card routing).
- `src/components/` — one component per file. `InProgressCard`, `OrderCard`, `PastOrderCard`, `ClaimCard` are the four baseline order-card variants; `DocsRejectedCard`, `PickupFailedCard`, `ResetFailedCard`, and `InvalidClaimCard` are takeover variants that supersede `ClaimCard` when the claim is blocked on a single customer action (re-upload evidence / confirm new pickup / unlink iCloud + share passcode / pay return shipping after an invalid claim). `WarrantyClaimCard` is a sibling of `ClaimCard` routed on `claim.type === 'warranty'` — same chrome, different post-QC tail (repair-and-ship-back). `ClaimCard` also serves `claim.type === 'compensation'` (refund-only, no pickup): it picks the status list via `claimStatusesFor(claim)` (4 dots, no Pickup) and shows "To be confirmed" when `claim.expectedRefund` is absent. `InvalidClaimCard` short-circuits to a private `CompensationClosedCard` ("Claim closed — no refund", no return-shipping gate) when the invalid claim is a compensation claim. `ClaimDetailsSheet` is `ClaimCard`'s details sheet (warranty-aware: hides the Refund destination row and renders a Return section for warranty claims; compensation-aware: shows the claim sub-type + description instead of Reason/Device-prep/Pickup rows, and a "To be confirmed" refund). `ClaimActionBanner` is a private child of `ClaimCard` rendered inside its expanded state when `claim.actionRequired` is set; the `See detailed tracking` dropdown (rendered when `claim.transitSubTimeline?.picked_up` is set on `ClaimCard`, or when `claim.shipBack?.awb` is set on `WarrantyClaimCard`) is a private helper inside each card. `UndoSnackbar` is a demo-only toast surfaced by `App.jsx` after the returns flow submits an in-session claim — auto-dismisses after 8s; Undo deletes the seeded claim from `submittedClaims`. See `docs/output/returns/claim_tracking.md` and `docs/output/warranties_compensations.md`. `WalletInfoTooltip` is shared anywhere "Revibe Wallet" is named — reuse it; it exports `REVIBE_WALLET_ICON`.
- `src/components/ClaimFlow/` — returns flow with four wired branches. `change_of_mind` and `issue` are 8 steps; `warranty` is 7 (Step 6 refund-method skipped); `compensation` is 5 (Steps 3/4/5 — device prep, packing, pickup — all skipped, since the customer keeps the device) — all driven by `visibleStepCount` / `visibleStepIndex` + NEXT/BACK/GO_TO_STEP step-overs in `flowReducer.js`. Step order: 1 ClaimType → 2 Reason/Issue/Compensation → 3 DevicePrep → 4 Packing → 5 Pickup → 6 Refund → 7 Review → 8 Confirmation. `ClaimFlow.jsx` mounts conditionally (no `open` prop — unmounting is what resets state) and on submit calls `onSubmitClaim(orderId, claim)` to seed the order at the App level. The Continue button is **never disabled** on any step (see Returns-flow mental model below): `ClaimFlow.handlePrimary` reads `stepError(state)` and, when non-null, dispatches `ATTEMPT` to surface the first unmet input inline instead of advancing. `flowReducer.js` owns the state shape, the soft-validation flag (`state.attempted`, set by `ATTEMPT` and cleared atomically by every step-changing action so the next step never flashes its errors), `stepError(state)` (the ordered "first unmet requirement" per step — single source of truth for what lights up red), and the legacy `canAdvance` validity predicate (retained but no longer gates the button). `InlineError.jsx` is the shared red message every step renders. Step 2 reuses `Step2IssueDetails` for both `issue` and `warranty`; `change_of_mind` uses `Step2Reason`; `compensation` uses `Step2Compensation` (sub-type picker from `compensationSubtypes.js` + evidence). The refund-method screen is skipped for warranty; compensation skips device prep / packing / pickup **and** shows no refund amount (always "confirmed by support after review") and no Review ack gate. Details in `docs/output/returns/change_of_mind.md`, `docs/output/returns/issue.md`, and `docs/output/warranties_compensations.md` §2 (warranty) + §3 (compensation).
- `src/data/orders.js` — mock orders. Field documentation lives in `docs/output/orders.md` §7 (top-level + status + tracking + timeline + product), `docs/output/cancellations.md` §7 (cancellation + refund), and `docs/output/returns/claim_tracking.md` §5 (claim object).
- `src/lib/statuses.js` — single source of truth: statuses, sub-statuses, header chips, banner copy/tone, `pickActiveOrderId`.
- `src/lib/returns.js` — single source of truth for returns: eligibility, refund math, fee rate, `ISSUE_WALLET_BONUS`, window, `generateClaimRef`.
- `src/lib/claims.js` — single source of truth for `ClaimCard` **and** `WarrantyClaimCard`: states, tone, copy helpers, `hasActiveClaim` / `isClaimRefunded` / `isWarrantyDelivered`. Hosts the refund pipeline (`CLAIM_STATUSES` + `claimToneFor` / `claimProgressIndex` / `claimPhaseTag`), the compensation pipeline (`COMPENSATION_CLAIM_STATUSES` — 4 states, no Pickup leg, reuses the refund status ids so tone/phase helpers apply unchanged), and the warranty pipeline (`WARRANTY_CLAIM_STATUSES` + `warrantyClaimToneFor` / `warrantyClaimProgressIndex` / `warrantyClaimPhaseTag`). `claimStatusesFor(claim)` resolves refund-vs-compensation to the right status list (warranty stays on its dedicated functions); `claimStatusHeadline` reads through it. Also hosts `CLAIM_TRANSIT_SUB_STATUSES` + `transitSubProgressIndex` (driving `ClaimCard`'s `See detailed tracking` dropdown when `claim.transitSubTimeline?.picked_up` is set — `WarrantyClaimCard`'s dropdown reuses `SHIPPING_SUB_STATUSES` from `lib/statuses.js` instead), `SUB_STATUS_LABELS` (customer-facing copy registry for each sub-status — kept for ops/internal references and any future inline surface), `CLAIM_SLAS` (SLA placeholders covering both refund and warranty-tail steps, used by the Step 4 detailed-timeline dropdown), `expectedCompletionFor(claimType)` (sums SLA `expectedHours` across the pipeline and returns `{ short, long, hours, date }` for the Step 4 "Expected by" headline + warranty Review / Confirmation surfaces), and `actionGateCopy` (banner copy for the three action gates).
- `src/lib/events.js` — `getHistoryEvents(order, mode)` drives the `HistoryThread` on layered cards.
- `src/data/journey.js` — `INITIAL_ORDER` + the `JOURNEYS` array (today: `happy_path`, `dynamic_edd`, `cancel_at_qc`, `claim_change_of_mind`, `claim_issue`, `claim_warranty`, `claim_compensation`). Replay journeys: each node is `{ id, label, trigger, event, apply(order), next? }`; branching is opt-in via `next: [...nodeIds]` (defaults to next-in-array; terminal nodes set `next: []` if not last). Sandbox journeys set `kind: 'sandbox'`, leave `nodes: []`, and are driven by their own hook + panel — see `useEddSandbox` / `EddSandboxPanel` below.
- `src/lib/journey.js` — `useJourney(journeyId)` hook. Path-based replay (`path: string[]`) so branched journeys resolve correctly. Exposes `validNext` / `advance` / `back` / `reset`, and a `kind: 'replay' | 'sandbox'` discriminator so `App.jsx` knows which panel to render.
- `src/lib/edd.js` — pure EDD model. Markets config (UAE / ZA / SA — working-day, today-buffer, ship buffers, weekend, stage SLAs), `workdayIntl` (Excel WORKDAY.INTL equivalent), `calculateEdd`, `orderStatus` (current/prev stage + SLA verdict + customer message + delivery_by + initial_promise), and exported `MSG_*` constants. 1:1 port of `brief/edd.py`.
- `src/lib/eddSandbox.js` — `useEddSandbox(journey)` hook for the Dynamic EDD sandbox journey. Owns the date/market inputs, computes status via `orderStatus`, synthesises an order with timeline + EDD fields populated and a `statusBanner: { tone, lead, body }` field that fully overrides `statusDescription`.
- `src/components/JourneyDevPanel.jsx` — bottom-right floating dev tool for replay journeys. Picker chip row (`flex-wrap`), step counter + back chevron, one `Next` button per `validNext()` option (customer-triggered styled outlined with a `via UI` chip; system-triggered solid brand). Exits journey mode via the header chip, not the panel.
- `src/components/EddSandboxPanel.jsx` — sibling of `JourneyDevPanel` rendered when `journey.kind === 'sandbox'`. Market select + four `<input type="date">` rows + an "actual delivered" toggle + a debug strip (stage / elapsed / SLA / message key / initial promise / EDD). Same chrome and journey-picker row as `JourneyDevPanel`.
- `brief/` — source screenshots + design-system reference.
- `docs/` — see `docs/README.md` for the per-feature doc map. Split into `docs/input/` (operational state-machine specs transcribed from drawio source) and `docs/output/` (prototype / UI feature specs).
- `CHANGELOG.md` — change history, phase by phase.

## Mental models

One-liners. Expand each in the linked per-feature doc if you need the why.

- **Journey mode.** Opt-in alternate demo (`?journey=<id>`) that replaces the eight-card showcase with a single order replayed through one lifecycle, driven by a floating `JourneyDevPanel`. Two purposes: immersive end-to-end demo, and the seed for a backend-event spec (detailed spec deferred until real-column mapping). Multi-journey + branching via top-level `JOURNEYS` array and `node.next: [...]`. Recipe + conventions: `docs/output/journey_backend_spec.md`.
- **Sandbox journeys.** Variant of journey mode for parameter-driven (not event-driven) demos. Journey sets `kind: 'sandbox'` + `nodes: []`; pairs with a dedicated hook (e.g. `useEddSandbox`) and panel (e.g. `EddSandboxPanel`). `App.jsx` calls both `useJourney` and the sandbox hook unconditionally and picks one based on `journey.kind`. The synthesised order feeds the existing card-routing tree unchanged. First example: `dynamic_edd` (interactive EDD + SLA-messaging playground; market / today / order / QC / shipped / delivered inputs). When the sandbox needs to inject banner copy that differs from the status-driven defaults, set `order.statusBanner: { tone, lead, body }` — `statusDescription` honours it as a full override (unlike `statusMessage`, which only overrides body).
- **Card routing.** `App.jsx` picks in order: `claim.docsRejection` → `DocsRejectedCard`; `claim.pickupFailure` → `PickupFailedCard`; `claim.resetFailed` → `ResetFailedCard`; `claim.invalidClaim` → `InvalidClaimCard` (which itself branches to a compensation closed-card when `claim.type === 'compensation'`); `claim.type === 'warranty'` + active → `WarrantyClaimCard` (In Progress); `hasActiveClaim` → `ClaimCard` (In Progress — also serves active `compensation` claims); `isWarrantyDelivered` → `WarrantyClaimCard` (Past); `isClaimRefunded` → `ClaimCard` (Past); else by `statusId`/`state`: `InProgressCard` (created/QC), `OrderCard` (shipped + in-flight cancelled), `PastOrderCard` (delivered without a claim, or cancelled-and-refunded). The first four are takeover cards — they replace the baseline card's surface while the claim is blocked on a single customer action and auto-cancels (or auto-closes) if ignored, ordered chronologically in the pipeline (docs at submit → pickup at collection → reset at QC wipe → invalid at QC verdict). `WarrantyClaimCard` is not a takeover; it's a sibling of `ClaimCard` for warranty claims with a repair-and-ship-back tail instead of a refund tail. Full tree in `docs/output/orders.md` §2.
- **Two-tier status.** `created → quality_check → shipped → delivered` drives the horizontal timeline. While `shipped`, `subStatusId` drives the vertical sub-timeline. No `delivered` sub-status. See `docs/output/orders.md` §4.
- **`state` is parallel to `statusId`.** `state` (`open`/`close`/`cancelled`) controls header chips, independent of progression. Cancelled orders keep the `statusId` they had at cancellation. Override: `delivered` always renders a green Delivered chip regardless of `state`.
- **Status banner tone resolution** (in `statusDescription(order)`):
  1. `state === 'cancelled'` → red "Refund in progress"
  2. `delayed === true` → orange "Taking longer than expected" (`OrderCard` only — `InProgressCard` keeps brand-purple for delayed QC, intentional product decision)
  3. otherwise → `STATUS_DESCRIPTIONS[statusId]` (or `shipped:{subStatusId}`)
  4. `statusMessage` overrides body only
- **Auto-expand.** Everything collapses by default. `pickActiveOrderId` returns the single most-in-flight order; only that card auto-expands, and only when no hero is showing above it (`!showHero`). Delivered never expands. Orders at `created` or `quality_check` never trigger the hero and never auto-expand — `InProgressCard` is their canonical surface, kept collapsed until the user opens it (the hero is built around courier/ETA data that doesn't exist yet at those states). `ClaimCard` does not currently participate. See `docs/output/orders.md` §5.2.
- **Whole header is the tap target.** Chevrons on `OrderCard` / `InProgressCard` are decorative.
- **Returns flow.** Overlay launched from `Raise a claim` on the delivered `PastOrderCard`. State in one `useReducer`; no session persistence (closing unmounts). Four wired branches: `change_of_mind` and `issue` are 8 steps and diverge at Step 2 (Reason vs Issue picker); `warranty` reuses the Issue picker on Step 2 and **skips Step 6** (refund-method) — 7 visible steps; `compensation` (keep the device, refund-only) uses its own Step 2 picker (`shipping_refund` / `charger`) and **skips Steps 3/4/5** (device prep, packing, pickup) — 5 visible steps, refund amount always deferred ("confirmed by support after review"), no Review ack gate. Progress bar driven by `visibleStepCount(claimType)`. Step 4 is **Packing** — a radio pick between *original Revibe box* and *sturdy post box*, each with a bullet-list of packing instructions. **Flow-wide soft validation: the Continue/Submit button is never grayed out on any step.** Clicking it with a required input still missing dispatches `ATTEMPT` (sets `state.attempted`), and the step surfaces its *first* unmet requirement — resolved by `stepError(state)`, one at a time — as an inline red highlight (`InlineError` + a danger border on the offending field) that scrolls itself into view, instead of advancing. The flag is reducer-owned and cleared by every step-changing action, so the next step never flashes errors before the user tries to advance. Step 7 (Review) is the original instance of this pattern: two acknowledgment cards (*factory reset* + *packed properly*) under Device preparation; clicking Submit with either unchecked flips the topmost unchecked card red (`AckCard`) instead of submitting. Eligibility + refund math in `src/lib/returns.js`; pipeline + expected-completion in `src/lib/claims.js`. Submit calls `onSubmitClaim(orderId, claim)` so `App.jsx` can stitch the seeded claim onto the order (in-session only — cleared on refresh; undoable via the `UndoSnackbar`). Chrome is deliberately checkout-style, **not** the order-card chrome family. See `docs/output/returns/change_of_mind.md`, `docs/output/returns/issue.md`, and `docs/output/warranties_compensations.md` §2.
- **History thread.** On layered cards (`ClaimCard`, cancelled `PastOrderCard` in `refund_pending` / `refunded`, `DeliveredOrderCard`), past events render as compact chips under the active hero; tapping a chip expands its detail inline (one open at a time). Derived in `src/lib/events.js` from `timeline` / `cancellationTimeline` / `cancellationRejection`. The active event lives in the hero and is excluded from the thread. Chip click handlers `stopPropagation` because the card header is one big tap target. See `docs/output/orders.md` §6.
- **Claim pipeline.** 5 states: `initiated` → `pickup` → `qc` → `refund_issued` → `refund_credited`. Tone progression warn (initiated/pickup/qc) → brand (refund_issued) → success (refund_credited). `hasActiveClaim` returns true while `claimStatusId !== 'refund_credited'`. Consolidated from 7 states in May 2026 — see `docs/output/returns/claim_tracking.md` §1. **Compensation** uses a parallel 4-state chain (`COMPENSATION_CLAIM_STATUSES`: `initiated`/Submitted → `qc`/Under review → `refund_issued` → `refund_credited`) — same ids minus the Pickup leg, so tone/phase helpers and `hasActiveClaim`/`isClaimRefunded` work unchanged; see `docs/output/warranties_compensations.md` §3.5.
- **Detailed claim tracking.** Two layers inside `ClaimCard`'s expanded state. `ClaimActionBanner` is the always-visible inline gate when `claim.actionRequired` is set (collection failed, missing documents, return-shipping payment). The Initiated-state hero carries a **scheduled-pickup strip** (CalendarClock + `scheduledPickup.date` + `scheduledPickup.slot`, MapPin + `pickupDetails.address`) — surfaces the courier window Revibe assigned alongside the address the customer submitted, only while `claimStatusId === 'initiated'`. `See detailed tracking` is an opt-in dropdown rendered between the dot strip and the history thread once `claim.transitSubTimeline.picked_up` is set — mirrors the outbound `HeroCard` dropdown in the light palette and uses the inverse-journey stops in `CLAIM_TRANSIT_SUB_STATUSES` (`picked_up` → `arrived_origin_hub` → `in_transit` → `arrived_revibe_hub`), with the current stop set by `claim.transitSubStatusId` and timestamps from `claim.transitSubTimeline`. Canonical reference: `docs/output/returns/claim_tracking.md` §2.2 + §2.3 + §4. Design history: `docs/output/claim_detailed_tracking.md`.
- **Invalid-claim ship-back.** When inspection determines a claim can't be approved, `InvalidClaimCard` takes over (danger tone) until the customer pays return shipping. Three internal demo states managed in-component: `action_needed` → `paid` (renders a fresh-order-style ETA hero + 4-step horizontal timeline driven by `claim.invalidClaim.returnShipment` — eyebrow becomes "Return from Claim RET-X" so lineage stays visible) or `declined` (muted "Claim closed — no refund" terminal with a warn-toned reversal CTA copy-locked to "I changed my mind and will pay for the shipment fee"). Inline edit mode on the delivery-details card via a "Change delivery details" button styled like `InProgressCard`'s "Change details". See `docs/output/returns/claim_tracking.md` §3.3.
- **Reset-failed iCloud unlink.** When QC tries to wipe an iOS device and Activation Lock is still on, `ResetFailedCard` takes over (danger tone) until the customer remotely removes the device from their iCloud account and shares their device passcode. Two internal demo states managed in-component: action-needed (numbered iCloud.com instructions card + confirmation toggle + masked 6-digit passcode field + ShieldCheck security note + danger-tinted Submit gated on both inputs) → submitted (warn-toned "Reset in progress" with a `What you sent` summary showing the unlink confirmation + last-two-digits-only masked passcode + replay-the-demo undo). Today routes regardless of `claim.devicePrep.os` but the copy is iOS-only. Submit is component-local — no `journey.advance` wiring yet, so journey mode advances the customer-triggered `claim_reset_details_received` node through the dev panel's `via UI` button. The journeys also model a one-shot retry loop (`claim_reset_retry_failed → claim_reset_retry_resubmitted`) before re-merging into the QC outcome fork. See `docs/output/returns/claim_tracking.md` §3.4.
- **Warranty pipeline.** Sibling of the refund pipeline, routed on `claim.type === 'warranty'` to `WarrantyClaimCard`. 6 states in `WARRANTY_CLAIM_STATUSES`: `initiated` → `pickup` → `qc` → `under_repair` → `ship_back` → `device_returned`. Head shared with refund flow; tail (`under_repair → ship_back → device_returned`) replaces the refund chain — no money changes hands, the same physical device comes back. Tone warn (initiated/pickup/qc) → brand (under_repair/ship_back) → success (device_returned). State-specific heroes: `RepairWindowStrip` (Wrench + estimated complete date) on `under_repair`; `InProgressCard`-style brand-gradient ETA hero on `ship_back`; `ReturnedStrip` (CheckCircle2 + delivered date) on `device_returned`. `See detailed tracking` dropdown reuses the **standard outbound `SHIPPING_SUB_STATUSES`** from `lib/statuses.js` (arrived in destination country → cleared customs → forwarded to third-party agent → out for delivery) — a repaired return reads with the same milestones as a fresh delivery. Gated on `claim.shipBack?.awb`, collapsed by default, brand-toned button to cue the tap. Spec: `docs/output/warranties_compensations.md` §2. Operational source: `docs/input/return_flow_warranty.md`.

## Conventions

- Default to **no comments**. Only write one when the *why* isn't obvious from the code.
- Component state stays in the component unless multiple components need it; lift to `App.jsx` and pass down (see filter state).
- New status / sub-status / state? Edit `src/lib/statuses.js` only — timeline, banner, header, chips are data-driven from there.
- Tailwind: prefer custom tokens (`brand`, `accent`, `success`, `progress`, `chip-*`, `searchBg`, `ink`, `muted`, `line`, `surface`) over arbitrary values. Slash-opacity (`bg-brand/10`) works on every token.
- "Need a screenshot for verification" → playwright at viewport 430×N, `deviceScaleFactor: 2`, `_snap.mjs` at the repo root.

## Gotchas

- **Tailwind name collisions.** `text-{name}` maps to either a fontSize or a color; defining `colors.page` and `fontSize.page` together silently produces white text. We've removed `colors.page`; don't reintroduce it. Same risk applies for any new token name.
- **Prototype links.** `CourierBanner` hardcodes the DHL Express tracking URL to a known-good test shipment (`tracking-id=3392654392`). Don't template it on `order.trackingNumber` — the demo orders have placeholder numbers that won't resolve.
- **Date filter is plumbed but inert.** All mock orders fall inside "Last 30 days," so the range dropdown has no visible effect.
- **Many things are visual placeholders.** Search, Wallet pill balance, profile menu, language toggle, "Download receipt" — all decorative. "Raise a claim" is wired for all four real branches (`change_of_mind`, `issue`, `warranty`, `compensation`). Submit seeds an in-session claim on the order (cleared on refresh, undoable via `UndoSnackbar`) — there's no backend, and seeding always lands on `initiated`. The `WarrantyClaimCard` has two hand-seeded mocks (89610 / 89580) exercising the `under_repair` / `ship_back` heroes; compensation has three (89630 under review, 89605 refunded, 89572 closed-invalid) for the surfaces submit-time seeding can't reach. Per-feature mocked-vs-prod lists in each `docs/output/*.md`.
