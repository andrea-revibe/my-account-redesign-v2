# Collaboration strategy

> How we keep agent collaboration on this repo effective and **token-efficient** as it grows toward a near-1:1 production copy of the My Account app. The throughline: agents should do **targeted reads**, not fan-out search, and should be able to reason about **cross-component / cross-flow impact** before changing anything.

## Operating principles (decided June 2026)

- **Generated + curated.** Anything derivable from source (dependency graph, symbol index) is generated by a script so it never goes stale; the "why" and the couplings a graph can't see are hand-curated. See `code_map.md`.
- **Mostly-additive churn.** Near-term work adds cards/flows/journeys on the existing `App.jsx` routing + `lib/` source-of-truth structure. Fine-grained maps are therefore a safe investment.
- **Protocol-only freshness.** No enforcement hooks/CI. The doc-update protocol in `CLAUDE.md` is the discipline; `npm run codemap` is the one command that regenerates the derived map.

## Roadmap

### P0 — Code map _(done)_

`docs/code_map.md` + `scripts/codemap.mjs` (`npm run codemap`). Navigation + impact layer: concept→file→line index, generated blast-radius/consumers table, and the string-contract coupling table (the cross-component dependencies no import graph shows). `CLAUDE.md` directs agents to read it before exploring. This is the foundation the items below build on.

### P1 — Split the read-whole hazards _(done)_

Three files dominated read cost; an agent needing one slice had to load thousands of lines. Split along existing boundaries, behavior-preserving and byte-verified against `HEAD` (build green, lint delta zero new errors, public exports unchanged):

| File | Before | After |
|---|--:|---|
| `data/journey.js` | 3188 | barrel (88) + `data/journeys/*` — one file per journey (`initialOrder`, `happyPath`, `cancellation`, `claimChangeOfMind`, `claimWarranty`, `claimIssue`, `claimCompensation`); largest 785 |
| `data/orders.js` | 1348 | barrel (19) + `data/orders/*` — `baseline` / `claims` (incl. takeovers) / `warranty` / `compensation`; spread order preserved = on-page order |
| `components/ClaimFlow/ResetGuideSheet.jsx` | 2477 | shell (865) + `resetGuideMocks.jsx` (1685, CSS-art screens) + `resetGuideAnim.js` (9, shared keyframes). A future pass could split the mocks by platform (iOS / macOS / Android) if that file becomes an edit hotspot.

### P2 — Cross-cutting mermaid diagrams

The per-feature docs already diagram individual flows. What's missing is the **connective** set — the diagrams a planning agent needs to see impact across flows. Each lives in (or is linked from) `code_map.md` so it's found before exploration:

1. **Card-routing decision tree.** The ~90-line precedence block in `App.jsx` (≈L285–375) as one mermaid flowchart: `claim.docsRejection` → `pickupFailure` → `resetFailed` → `invalidClaim` → `type==='warranty'` → `hasActiveClaim` → `isWarrantyDelivered` → `isClaimRefunded` → baseline by `statusId`/`state`. A plan agent touching routing reads the diagram, not the file.
2. **Unified claim-lifecycle state diagram.** All four pipelines on one canvas — refund (`CLAIM_STATUSES`), compensation (`COMPENSATION_CLAIM_STATUSES`), warranty (`WARRANTY_CLAIM_STATUSES`), plus the four takeover detours — annotated with which card renders at each state.
3. **Returns submit→seed→project→render data-flow.** `ClaimFlow.onSubmitClaim` → `App.submittedClaims` → projection over `ORDERS` (≈L204) → card routing. This is the flow most likely to confuse, because the coupling is a runtime projection, not an import.

Why it saves tokens: one diagram read replaces multi-file reconstruction of control flow.

### P3 — Slim the always-loaded context

`CLAUDE.md` (~27KB) loads on **every** invocation, so every token shaved is saved on every turn. Its dense "Where things live" / "Mental models" sections now overlap with `code_map.md` (the *where*) and `docs/output/*` (the *why*). Refactor `CLAUDE.md` into a lean **router**:

- Keep: scope, where-to-work, stack, conventions, gotchas, the doc-update protocol, and mental-model **one-liners** that aren't captured elsewhere.
- Move/trim: per-file "Where things live" prose → `code_map.md`; deep mental-model paragraphs → the relevant `docs/output/*` doc, leaving a one-line pointer.

Treat this as a careful, reviewed edit (judgment about what's load-bearing), not a mechanical cut. Keep `CHANGELOG.md` out of the hot path — it is not auto-loaded; don't add references that pull it in.

### P4 — Subagent discipline + optional auto-freshness

- **Explore sparingly, seed it narrow.** Standing guidance (encode in `CLAUDE.md`): prefer `code_map.md` + targeted reads; spawn an `Explore`/`Plan` agent only for genuinely-unknown locations or broad sweeps, and when you do, **pass the relevant code_map rows** in the prompt so it starts from a narrow frontier instead of re-discovering structure.
- **Plan agents must state blast radius.** Before proposing a multi-file change, a Plan agent looks up the target in code_map's consumers + coupling tables and lists the affected files in its plan.
- **Optional auto-freshness (opt-in).** Wire `codemap` into a `prebuild` npm script so `npm run build` regenerates the map — zero-discipline freshness without a git hook. Off by default; enable on request.
