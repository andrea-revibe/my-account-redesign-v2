# Collaboration strategy

> How we keep agent collaboration on this repo effective and **token-efficient** as it grows toward a near-1:1 production copy of the My Account app. The throughline: agents should do **targeted reads**, not fan-out search, and should be able to reason about **cross-component / cross-flow impact** before changing anything.

## Operating principles (decided June 2026)

- **Generated + curated.** Anything derivable from source (dependency graph, symbol index) is generated by a script so it never goes stale; the "why" and the couplings a graph can't see are hand-curated. See `code_map.md`.
- **Mostly-additive churn.** Near-term work adds cards/flows/journeys on the existing `App.jsx` routing + `lib/` source-of-truth structure. Fine-grained maps are therefore a safe investment.
- **Protocol-only freshness.** No enforcement hooks/CI. The doc-update protocol in `CLAUDE.md` is the discipline; `npm run codemap` is the one command that regenerates the derived map.

### P2 тАФ Cross-cutting mermaid diagrams _(done)_

`docs/output/diagrams.md` (linked from a `Cross-cutting diagrams` section in `code_map.md`) holds the three diagrams below, each with a "read before / source" pointer back to the code. Hand-maintained (no generator) тАФ update the matching diagram when routing precedence, a claim pipeline, or the submitтЖТrender path changes.

The per-feature docs already diagram individual flows. What's missing is the **connective** set тАФ the diagrams a planning agent needs to see impact across flows. Each is found via `code_map.md` before exploration:

1. **Card-routing decision tree.** `App.jsx`'s routing as one mermaid flowchart тАФ modelled faithfully as the two-stage `isOpen(order)` partition (In progress vs Past) then the precedence ladder inside each section (`docsRejection тЖТ pickupFailure тЖТ resetFailed тЖТ invalidClaim тЖТ warranty тЖТ hasActiveClaim тЖТ in-flight cancellation тЖТ created/QC тЖТ OrderCard`; past: `isWarrantyDelivered тЖТ isClaimRefunded тЖТ PastOrderCard`). A plan agent touching routing reads the diagram, not the file.
2. **Unified claim-lifecycle state diagram.** All four pipelines on one canvas тАФ refund (`CLAIM_STATUSES`), compensation (`COMPENSATION_CLAIM_STATUSES`), warranty (`WARRANTY_CLAIM_STATUSES`), plus the four takeover detours тАФ annotated with which card renders at each state.
3. **Returns submitтЖТseedтЖТprojectтЖТrender data-flow.** `ClaimFlow.onSubmitClaim` тЖТ `App.submittedClaims` тЖТ projection over `ORDERS` (тЙИL204) тЖТ card routing. This is the flow most likely to confuse, because the coupling is a runtime projection, not an import.

Why it saves tokens: one diagram read replaces multi-file reconstruction of control flow.

### P3 тАФ Slim the always-loaded context

`CLAUDE.md` (~27KB) loads on **every** invocation, so every token shaved is saved on every turn. Its dense "Where things live" / "Mental models" sections now overlap with `code_map.md` (the *where*) and `docs/output/*` (the *why*). Refactor `CLAUDE.md` into a lean **router**:

- Keep: scope, where-to-work, stack, conventions, gotchas, the doc-update protocol, and mental-model **one-liners** that aren't captured elsewhere.
- Move/trim: per-file "Where things live" prose тЖТ `code_map.md`; deep mental-model paragraphs тЖТ the relevant `docs/output/*` doc, leaving a one-line pointer.

Treat this as a careful, reviewed edit (judgment about what's load-bearing), not a mechanical cut. Keep `CHANGELOG.md` out of the hot path тАФ it is not auto-loaded; don't add references that pull it in.

### P4 тАФ Subagent discipline + optional auto-freshness

- **Explore sparingly, seed it narrow.** Standing guidance (encode in `CLAUDE.md`): prefer `code_map.md` + targeted reads; spawn an `Explore`/`Plan` agent only for genuinely-unknown locations or broad sweeps, and when you do, **pass the relevant code_map rows** in the prompt so it starts from a narrow frontier instead of re-discovering structure.
- **Plan agents must state blast radius.** Before proposing a multi-file change, a Plan agent looks up the target in code_map's consumers + coupling tables and lists the affected files in its plan.
- **Optional auto-freshness (opt-in).** Wire `codemap` into a `prebuild` npm script so `npm run build` regenerates the map тАФ zero-discipline freshness without a git hook. Off by default; enable on request.

### P5 тАФ Docs reorg + review strategy _(done)_

The flat `docs/` folder mixed five read-cadences in one space. Reorganised by cadence so the daily working set stays small:

- **`handoff/backend-mapping/`** тАФ the two `db_explorer_*` briefs (read only by the separate backend-mapping project) moved out of the working set and renamed `context.md` / `system_prompt.md`.
- **`process/`** тАФ this doc moved here (docs-about-the-docs).
- **Deleted** `output/claim_detailed_tracking.md` тАФ design history of a removed disclosure; its live tables already live in `output/returns/claim_tracking.md` ┬з4, and the reasoning is in git history.
- `input/` + `output/` names kept (light touch тАФ avoids re-pointing the cross-link web and the external project's references).

The **review strategy** (tiered loading + freshness frontmatter + trigger-based verification + opt-in confidence pass) is documented in `docs/README.md` ┬з "Doc tiers & review strategy". Principle: separate docs by how often they load, then verify on-trigger rather than on a schedule тАФ no cold docs in the working set, no idle audit sweeps.
